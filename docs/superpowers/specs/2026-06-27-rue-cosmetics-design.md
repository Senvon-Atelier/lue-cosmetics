# Rue Cosmetics — Case Study Design

**Date:** 2026-06-27
**Status:** Draft for review (revised after external review)
**Goal:** Turn the existing Rue Cosmetics frontend mockup (in `casestud/Rue/`) into a fully functional e-commerce case study, deployed to the user's Hetzner server behind Caddy.

## Naming conventions used throughout this spec

- **Brand / product name:** Rue Cosmetics
- **Project folder:** `casestud/ruecosmetics/`
- **Go module:** `github.com/oti-adjei/ruecosmetics` (or equivalent; finalized at scaffold time)
- **Existing mockup folder being ported from:** `casestud/Rue/`

## 1. Purpose & Success Criteria

Rue Cosmetics is a Ghana-based cosmetics & wellness e-commerce concept. It was originally designed for potential clients who did not engage. It is being repurposed as a **public case study**: a working website visitors can sign up to, browse, add to cart, check out (in payment-gateway test mode), and explore as either a fresh customer or as one of several pre-seeded demo accounts.

**Success means:**

- A visitor can land on the homepage, browse the catalog, add items, register an account (or log in to a seeded account), check out via Paystack test mode, and see the resulting order in their account dashboard.
- An admin user can log in to the admin dashboard, see real orders/users/revenue from the database, and update an order's fulfillment status.
- The system is built and presented in a way that is impressive to a reviewer evaluating the user's portfolio: real database, real auth, real payments (in test mode), real email, type-safe end-to-end with a single source of truth, RBAC enforced in depth, atomic commerce operations.
- Deployed at `rue.example.com` (frontend) and `api.rue.example.com` (backend) on the user's existing Hetzner host behind their existing Caddy setup.

**Explicit non-goals (v1):**

- Real money / production-mode payments.
- Admin CRUD for the product catalog (catalog is DB-backed, seeded from `data.js`, mutable in future versions but not via UI in v1).
- Customer-written product reviews (rating + review count are seeded fields only).
- Coupon / discount code system.
- Return/RMA workflow (the `refunded` and `cancelled` order statuses exist; the workflow does not).
- A blog/CMS UI for editing posts (blog content lives as static files imported by Vite).
- Multi-currency (everything is GHS).
- Metrics / tracing observability stack.
- Background job queue (emails sent inline; future work).
- In-memory caching (Postgres reads are fast enough for v1; future work).

## 2. Stack Decisions

| Layer | Choice | Reason |
|---|---|---|
| Frontend framework | **Vite + React 18 + TanStack Router + TanStack Query** | Cleanest split with a separate Go backend; TanStack ecosystem showcases modern React DX. |
| Frontend styling | **Tailwind CSS v4** | Existing mockup's palette + typography mapped into Tailwind theme tokens. |
| Frontend API client | **Orval** — generates client + Zod schemas + React Query hooks from the backend's OpenAPI spec | Single source of truth (Go → OpenAPI → Orval) eliminates handwritten clients and schema drift. |
| Frontend validation | **Zod** (generated by Orval; handwritten only for form-only schemas with no API counterpart) | Best ecosystem fit with BetterAuth + TanStack Form. |
| Frontend package manager | **pnpm** | User preference. |
| Auth | **BetterAuth (Go) with admin/roles plugin** | Handles email/password + Google OAuth + session cookies + role storage in one library. |
| Auth methods | **Email/password + Google OAuth** | Demonstrates breadth without the email-sending dependency of magic links. |
| Backend language | **Go 1.22+** | Stronger portfolio showcase than a single TS stack. |
| Web framework | **chi** | Idiomatic Go, minimal, stdlib-flavored. |
| DB driver | **pgx/v5** (`jackc/pgx`) with `pgxpool` | Faster than `database/sql`, first-class Postgres types (arrays, JSONB, COPY). |
| DB query layer | **sqlc** generating against the `pgx/v5` driver | Typed Go from plain SQL — no ORM, no injection paths. |
| OpenAPI generation | **`swaggo/swag`** annotations on chi handlers | Source of truth for the wire contract; consumed by Orval. |
| Migrations | **goose** | Simple CLI, supports both SQL and Go migrations. |
| Backend validation | **`go-playground/validator`** | Struct-tag-driven, standard in Go ecosystem. |
| Backend config | **Typed `Config` struct populated via `envconfig`** | One place to change env vars; testable services. |
| Database | **Postgres 16** | Locally installed on the Hetzner host (not managed). |
| Payments | **Paystack** | Native GHS + Mobile Money (MTN MoMo, Vodafone Cash). Test mode for the case study. |
| Email | **Resend** in allowlist mode (real outbound to allowlisted addresses only); **Mailpit** in local dev | Real-shaped flow without deliverability risk. |
| Repo structure | **Monorepo** (`frontend/` + `backend/`) | Easiest mental model for reviewers. |
| Deploy | **Caddy reverse proxy → static frontend + Go binary (systemd)** | Uses the user's existing infrastructure. |
| CI | **GitHub Actions** | Standard. |

### 3.1 Backend layering

Three-layer architecture for domains with real business logic; two layers for domains that are pure reads.

```
HTTP handler (chi)
    │  parses + validates request, calls service
    ▼
Service                          ← business logic, transactions, side effects
    │  uses one or more repositories within a tx
    ▼
Repository (sqlc-generated)      ← typed SQL queries
    │
    ▼
Postgres
```

**Domains that get a full service layer** (because they have transactions, multiple repositories, or external side effects):

- `auth` — wraps BetterAuth + the application's role checks
- `cart` — cart merge, pricing, totals
- `orders` — checkout, payment marking, state transitions
- `payments` — Paystack client + webhook verification + verify-poll
- `email` — Resend client + Mailpit dev provider
- `admin` — order-status transitions, stats aggregation

**Domains that stay handler → repository** (pure reads, no business logic worth abstracting):

- `catalog` (products, categories, brands)
- `wishlist` (single-table CRUD)
- `addresses` (single-table CRUD)

### 3.3 Local dev

- `docker compose up` → Postgres 16 on `:5432`, Mailpit on `:8025`.
- Backend: `air` (hot reload) on `:8080`, reads `.env` into the typed `Config`.
- Frontend: `pnpm dev` on `:5173`, Vite proxies `/api/*` → `:8080`.
- API contract: `make openapi` regenerates `backend/docs/openapi.json` (swaggo) and runs Orval to refresh `frontend/src/lib/api/generated/`. CI fails if either is stale.

### 3.4 Production

- Single Hetzner host, single Postgres instance.
- Backend deployed as a single Go binary + systemd unit (`rue-api.service`), reads `/etc/rue/api.env`.
- Frontend deployed as static `dist/` served by Caddy directly.
- Two Caddy sites: `rue.example.com` (static) and `api.rue.example.com` (reverse proxy to `127.0.0.1:8080`).
- Both subdomains share an eTLD+1 so the BetterAuth session cookie is valid across them.

## 4. Data Model

UUID primary keys via `gen_random_uuid()` (pgcrypto). All monetary values stored as `BIGINT` in **pesewas** (1 GHS = 100 pesewas) to avoid float drift.

### 4.1 Tables

**Identity & auth** (managed by BetterAuth schemas + admin plugin)

- `users` — `id, email, name, image, email_verified, created_at, updated_at`
- `accounts` — OAuth provider links (Google) + credential rows for email/password
- `sessions` — `id, user_id, token, expires_at, ip, user_agent`
- `verification` — magic tokens for email verify + password reset
- `user_roles` — `user_id, role` where role ∈ {`customer`, `admin`} (BetterAuth admin plugin)

**Catalog** (DB-backed, seeded from `Rue/src/data.js` via `seed/data/products.json`; UI-immutable in v1, mutable in future versions)

- `categories` — `id, slug, label, sort_order`
- `brands` — `id, name, slug`
- `products` — `id, slug, name, brand_id, category_id, price_ghs_minor, was_price_ghs_minor (nullable), tone, size, rating, review_count, tags text[], image_path, created_at`

  **`image_path` contract:** stored as a **relative path** like `products/cream-serum.jpg`. Frontend prepends its own origin (production: `https://rue.example.com/`; dev: `http://localhost:5173/`) and serves the file from `frontend/public/products/`. No absolute URLs in the DB.

**Commerce**

- `carts` — `id, user_id (nullable), guest_token (nullable), created_at, updated_at`. CHECK: exactly one of `user_id`/`guest_token` is non-null.

  **`guest_token` contract:** UUIDv4 generated by the backend on first `GET /cart` from an unauthenticated, untokenized client. Returned in the response body as `guest_token` AND set as a non-HttpOnly cookie `rue_guest_cart` (SameSite=Lax, 30-day expiry) so subsequent requests carry it automatically. Frontend mirrors it into localStorage under `rue.guest_cart` for use during the post-login merge call. No signing — the token is an opaque capability; possession = ownership of the cart.

- `cart_items` — `id, cart_id, product_id, qty, unit_price_ghs_minor` (price captured at add-time)
- `orders` — `id, user_id, status, subtotal_ghs_minor, shipping_ghs_minor, total_ghs_minor, paystack_reference (unique, format "RUE-{8-char short uuid}"), paystack_transaction_id (nullable, set on first paid/verify), shipping_address jsonb, created_at, updated_at`. Status ∈ {`pending`, `paid`, `fulfilled`, `shipped`, `delivered`, `cancelled`, `refunded`}.
- `order_items` — `id, order_id, product_id, qty, unit_price_ghs_minor, product_name_snapshot, product_brand_snapshot, product_image_snapshot` (explicit snapshot columns; historical orders survive catalog changes)
- `addresses` — `id, user_id, label, line1, line2 (nullable), city, region, phone, is_default boolean`
- `wishlist_items` — `(user_id, product_id)` composite PK, `created_at`

### 4.2 Static content (NOT in DB; frontend repo files)

Per external review #5: moved to the frontend. Vite imports them directly; no HTTP roundtrip.

- `frontend/src/content/blog/*.md` — blog posts (sourced from `Rue/src/data.js`)
- `frontend/src/content/testimonials.json` — testimonials
- `frontend/src/content/legal/{privacy,terms,returns}.md`

The backend has **no** `/blog`, `/testimonials`, or `/legal` endpoints.

### 4.3 Runtime config (backend, file-based)

- `backend/seed/config/shipping_config.json` — `{flat_rate_ghs_minor: 2500, free_over_ghs_minor: 50000}` (GHS 25 flat, free over GHS 500). **Loaded once at process start by `shipping.Service`, cached in memory.** Changes require restart. Single source of truth — every shipping calculation (cart pricing, `/checkout/init`, `/shipping/quote`) reads from the same in-memory copy.

### 4.4 Cart-merge rule

On login, the frontend reads `rue.guest_cart` from localStorage and POSTs it to `/cart/merge`. The backend transactionally:

1. Looks up the guest cart by token.
2. For each guest item: upserts into the user's cart by `product_id` — sum quantities; keep the **user's existing** `unit_price_ghs_minor` if both rows exist (the user's snapshot wins).
3. Deletes the guest cart row.
4. Clears the `rue_guest_cart` cookie via Set-Cookie response header.

Frontend then deletes `rue.guest_cart` from localStorage.

### 4.5 Guest cart lifecycle

- **No session, no token, first `GET /cart`** → backend creates a new guest cart, returns token via body + cookie.
- **No session, token present** → backend serves that cart.
- **Login while holding a guest token** → frontend POSTs `/cart/merge`; guest cart is consumed and deleted (see 4.4).
- **Logout** → BetterAuth clears the session cookie. The user's persistent cart stays in DB attached to their user_id. No new guest cart is created until next anonymous `GET /cart`.
- **No expiry on guest tokens in v1.** Stale guest carts accumulate; a cleanup job is documented as future work.

## 5. API Surface

REST under `/api/v1`. JSON in, JSON out. Wire contract defined by `swaggo/swag` annotations on chi handlers → `backend/docs/openapi.json` → Orval-generated frontend client. Drift fails CI.

### 5.1 Auth ownership boundary

| Concern | Owner |
|---|---|
| User table schema, accounts, sessions, verification tokens | BetterAuth |
| Login, signup, OAuth (Google), password reset, email verification endpoints under `/auth/*` | BetterAuth |
| Password hashing | BetterAuth (argon2) |
| Role storage (`user_roles`) and role plugin | BetterAuth admin plugin |
| Session cookie lifecycle (set, refresh, clear) | BetterAuth |
| Custom user profile fields beyond BetterAuth's defaults | Application (`auth` service + dedicated tables if needed) |
| RBAC middleware (`RequireSession`, `RequireRole`) | Application |
| Authorization checks inside business logic (`MustBeAdmin`, row-level scoping) | Application |
| Account dashboard pages, address management, order history | Application |
| Allowlist email behavior + auto-verifying non-allowlisted users | Application (interposes between BetterAuth and the email service) |

### 5.2 Endpoints

**BetterAuth (mounted at `/api/v1/auth/*`)**

- Standard BetterAuth routes: signup, login, logout, oauth/google, password reset, verify email, session. Schemas determined by BetterAuth — exposed to the frontend through BetterAuth's React client, not through Orval.

**Catalog (public)**

- `GET /products` — query: `category`, `brand`, `tag`, `q`, `sort` (allowlisted enum), `page`, `limit`
- `GET /products/:slug`
- `GET /categories`
- `GET /brands`

**Cart (public — uses session or guest token)**

- `GET /cart` — returns `{items, subtotal_ghs_minor, shipping_cost_ghs_minor, free_shipping_remainder_ghs_minor, total_ghs_minor}`; creates a guest cart on first call if no auth + no token (see 4.5)
- `POST /cart/items` — `{product_id, qty}`
- `PATCH /cart/items/:id` — `{qty}`
- `DELETE /cart/items/:id`
- `POST /cart/merge` — body: `{guest_token}`. Auth required.

**Shipping (public)**

- `GET /shipping/quote?subtotal=NNN` — returns `{flat_rate_ghs_minor, free_over_ghs_minor, applied_cost_ghs_minor, free_shipping_remainder_ghs_minor}`

**Wishlist (auth required)**

- `GET /wishlist`
- `POST /wishlist/:product_id`
- `DELETE /wishlist/:product_id`

**Account (auth required)**

- `GET /me` · `PATCH /me`
- `GET /me/addresses` · `POST /me/addresses` · `PATCH /me/addresses/:id` · `DELETE /me/addresses/:id`
- `GET /me/orders` · `GET /me/orders/:id`

**Checkout & payments**

- `POST /checkout/init` — auth required. Body: `{address_id, shipping_method}`. **Transactional.** Re-prices cart from DB, snapshots line items, inserts `orders` + `order_items` with status `pending`, generates `paystack_reference` (format `RUE-{8-char}`), calls Paystack `transaction/initialize`. Returns `{authorization_url, reference}`.
- `POST /webhooks/paystack` — public; signature-verified via HMAC-SHA512 against `PAYSTACK_SECRET`. **Calls `OrderService.MarkPaid(ctx, reference, transactionID)` — the same method the verify endpoint calls. Idempotent.**
- `GET /checkout/verify/:reference` — auth required. If local order status is still `pending`, backend calls Paystack's verify API, then calls `OrderService.MarkPaid(...)`. Both webhook and verify paths converge on this one method.

**Admin (admin role required)**

- `GET /admin/orders` — paginated, filterable by status
- `GET /admin/orders/:id`
- `PATCH /admin/orders/:id/status` — `paid → fulfilled → shipped → delivered`, or `→ cancelled`, or `→ refunded`. State machine validates transitions.
- `GET /admin/users` · `GET /admin/users/:id`
- `GET /admin/stats` — revenue, order counts by status, top products. **Computed live in v1; caching intentionally deferred (Section 15).**

**Cross-cutting:**

- Non-public routes go through `auth.RequireSession` middleware that validates the BetterAuth session cookie and injects `userID` + `role` into `r.Context()`.
- Admin routes go through an additional `auth.RequireRole("admin")` middleware **and** each handler calls `auth.MustBeAdmin(ctx)` at the top (redundant by design; see 10.3).
- Request bodies validated with `go-playground/validator` struct tags; failures return `400` with field-keyed error map.
- Errors follow a single shape: `{"error": {"code": "...", "message": "...", "fields": {...}?}}`.
- Rate limit on `/auth/*`: 10 req/min/IP via `httprate`.
- CORS: allowlist of origins read from `CORS_ORIGINS` env var (comma-separated). Production: `https://rue.example.com`. Dev: `http://localhost:5173`. Configurable so OAuth callbacks or additional subdomains can be added without code changes.

## 6. Auth, Payments, Email Flows

### 6.1 Auth

- BetterAuth Go SDK + admin/roles plugin owns: signup, login, OAuth (Google), email verification token issuance, password reset.
- The application interposes on email delivery (see 6.3) to handle the allowlist.
- Session: HTTP-only cookie set by BetterAuth, SameSite=Lax, 30-day rolling. Cookie scoped to eTLD+1 so it works on both subdomains.
- Frontend uses BetterAuth's React client for session state. Protected routes use TanStack Router's `beforeLoad` to redirect unauthenticated users; admin routes additionally check `role === 'admin'`.

### 6.2 Checkout (Paystack)

1. Frontend POSTs `/checkout/init` with `{address_id, shipping_method}`.
2. Backend opens a transaction (`pgxpool.BeginTx`):
   - Re-prices the cart from DB (client prices ignored).
   - Inserts `orders` row with status `pending` and `paystack_reference = "RUE-" + short_uuid()`.
   - Snapshots line items into `order_items` (name, brand, image, qty, unit price).
   - Commits.
3. Backend calls Paystack `transaction/initialize` with `amount = total_ghs_minor`, `reference = order.paystack_reference`. (External call is outside the DB transaction — atomicity of the DB write is what matters; if Paystack fails, the order stays `pending` and is documented as a known dead state in v1.)
4. Backend returns `{authorization_url, reference}`. Frontend redirects.
5. After payment, Paystack redirects to `rue.example.com/checkout/return?reference=...`. Frontend polls `GET /checkout/verify/:reference` every 2s for up to 30s.
6. In parallel, Paystack POSTs `/webhooks/paystack`.
7. **Both webhook and verify-poll converge on `OrderService.MarkPaid(ctx, reference, transactionID)`:**
   ```
   MarkPaid:
     BEGIN
       SELECT order FOR UPDATE WHERE paystack_reference = $1
       IF status != 'pending': COMMIT and return (idempotent no-op)
       UPDATE orders SET status='paid', paystack_transaction_id=$2, updated_at=now()
       DELETE FROM cart_items WHERE cart_id = (SELECT id FROM carts WHERE user_id = order.user_id)
     COMMIT
     enqueue confirmation email (in v1: send inline; future: queue)
   ```
8. Mobile money flows through the same Paystack hosted page (provider selection happens on Paystack's side).

**Failure modes:**

- Webhook never arrives → verify-poll catches it via Paystack verify API.
- User abandons Paystack page → order stays `pending`. No automatic expiry in v1 (future work).
- Cart changes between init and webhook → irrelevant; the order is snapshotted at init time.
- Paystack init call fails after order row written → order stays `pending` with no `authorization_url` ever returned to client; nightly cleanup is future work.

### 6.3 Email (Resend, allowlist mode)

- Env var `EMAIL_ALLOWLIST` = comma-separated addresses that receive real mail.
- `email.Service.Send(to, template, data)`:
  - If `to ∈ allowlist`: send via Resend (prod) or Mailpit (dev).
  - If `to ∉ allowlist`: log payload to stdout, return success without calling Resend.
- **Email-verification special case:** when a user signs up with a non-allowlisted email, the application calls BetterAuth's "mark verified" API immediately after signup, server-side. The user is never shown a "Please verify your email" screen. Allowlisted users go through the normal verify flow.
- Templates (Go `html/template`): signup welcome + verify (allowlisted only), password reset, order confirmation, order shipped.
- In dev, `EMAIL_PROVIDER=mailpit` swaps Resend for SMTP-to-Mailpit (visible at `localhost:8025`).
- Emails are sent **inline** in v1 (no queue). If Resend is slow, the calling handler is slow. Future work: queue + worker.

## 7. Seed Data (declarative)

A `cmd/seed/main.go` program populates the DB from a clean state, reading from `backend/seed/data/*.json`. The seed binary is a thin loader; the data lives in editable files.

```
backend/seed/data/
├── products.json     # 25 entries, sourced from Rue/src/data.js
├── categories.json
├── brands.json
├── users.json        # demo accounts with hashed passwords + role
├── addresses.json    # keyed by demo user email
├── orders.json       # demo orders with line items
└── wishlists.json    # keyed by demo user email
```

**Demo user shapes** (printed on seed run with their plain-text demo passwords):

- Customer with 3 paid + 1 delivered orders, populated wishlist, 2 addresses.
- Customer with delivered history (no active orders).
- Customer with only wishlist items.
- Customer in fresh/empty state.
- Admin user.

**Seed conflict semantics:**

- **Catalog (`categories`, `brands`, `products`)**: `INSERT … ON CONFLICT (slug) DO UPDATE` — re-running picks up edits to JSON.
- **Demo users**: `INSERT … ON CONFLICT (email) DO UPDATE SET name=..., role=...` — re-running refreshes demo data. **Exception:** if a row with the demo email exists AND was not created by a previous seed run (heuristic: `accounts.provider != 'seed-marker'`), the seed logs a warning and skips that user. This protects against the case where a real signup grabbed a demo email before the first seed.
- **Demo orders / addresses / wishlists**: deleted and re-created on every seed run (scoped to the demo user IDs only). Real users' data is untouched.
- The seed records `provider='seed-marker'` on demo accounts so it can identify its own rows on re-run.

## 10. Quality Guarantees

### 10.1 Type safety (continuous, single source of truth)

```
chi handler (Go) with swaggo annotations
        ↓
backend/docs/openapi.json
        ↓
Orval (orval.config.ts)
        ↓
frontend/src/lib/api/generated/
    ├── client.ts        (typed fetch wrappers)
    ├── hooks.ts         (React Query hooks)
    ├── schemas.ts       (Zod schemas, used at runtime)
    └── types.ts         (TS types)
```

- **Backend:** `go vet` + `staticcheck` + `golangci-lint` in pre-commit and CI. sqlc generates typed Go from SQL — query results are never `interface{}`.
- **Frontend:** `tsc --noEmit` in pre-commit and CI. `tsconfig` has `"strict": true` and `"noUncheckedIndexedAccess": true`. ESLint with `@typescript-eslint/recommended-type-checked`.
- **Wire boundary:** every API response is parsed through an Orval-generated Zod schema at runtime. No handwritten Zod schemas for endpoints that exist in the OpenAPI spec.
- **Drift check:** CI runs `make openapi` and fails if the generated `openapi.json` or the Orval output differs from what's committed. Forces regeneration before merge.
- Pre-commit (lefthook) runs `gofmt`, `golangci-lint`, `tsc --noEmit`, `eslint`, `prettier` (with `prettier-plugin-tailwindcss` for class sorting).

### 10.2 SQL injection — structurally impossible

- All queries flow through sqlc-generated functions using `pgx` parameterized queries.
- Dynamic sort/filter columns on `/products` and `/admin/orders` use an **allowlist enum** mapped to fixed `ORDER BY` clauses. Unknown sort values return 400.
- CI grep rule: `grep -rE "fmt\.Sprintf.*(SELECT|INSERT|UPDATE|DELETE)" backend/` must return nothing.

### 10.3 RBAC — defense in depth (backend authoritative)

- **Frontend hides admin nav for non-admins. This is UX, not security. The backend is the only authoritative authority.**
- **Layer 1 — router group:** admin routes mounted under a chi group with `RequireRole("admin")` middleware.
- **Layer 2 — handler:** every admin handler calls `auth.MustBeAdmin(ctx)` at the top. Redundant by design — survives router refactors and accidental route additions outside the group.
- **Layer 3 — query scope:** `/me/*`, `wishlist`, `cart` queries always include `WHERE user_id = $1`. The `user_id` is taken from session context, **never** from request body or URL.
- **Tests:** an integration test matrix asserts customer-session → admin route → 403 for every admin endpoint, and customer-A → customer-B's resources → 404/403 (IDOR coverage).

### 10.4 Atomicity — commerce operations are transactional

Every commerce service method that mutates more than one row runs inside `pgxpool.BeginTx`:

- `cart.MergeService.Merge` — guest items + user items + delete guest cart.
- `orders.CheckoutService.Init` — order row + order_items rows.
- `orders.OrderService.MarkPaid` — order status update + cart clear.
- `admin.OrderService.UpdateStatus` — order status update (single row, but uses tx for `SELECT … FOR UPDATE` + audit log if added later).

A `db.WithTx(ctx, fn)` helper enforces commit/rollback semantics so handlers never see a half-applied state.

## 11. Testing

- **Backend unit:** table-driven tests for pure functions (cart pricing, shipping calc, cart merge logic at the service level with a mocked repository, order state-machine transitions).
- **Backend integration:** `testcontainers-go` spins a real Postgres per test package. Covers happy paths, the RBAC matrix, Paystack webhook signature verification (known-good + tampered payload), idempotent `MarkPaid` (calling it twice has no side effect), and the verify-poll + webhook race (both arrive, only one mutation).
- **Frontend unit:** Vitest for utilities + Orval-generated Zod schema round-trips.
- **Frontend component:** Testing Library for stateful components (cart drawer, checkout form, address picker).
- **E2E:** one Playwright happy-path: signup → browse → add to cart → checkout (Paystack test card) → see order in account. Runs in CI against a docker-compose stack.

## 12. CI/CD

- **GitHub Actions:**
  - On push: lint, type-check, unit tests (frontend + backend), OpenAPI/Orval drift check.
  - On PR to main: integration tests + E2E.
  - On merge to main: build artifacts (backend binary + frontend `dist/`).
- **Deploy:** manual `make deploy` target — `scp` binary + `dist/` to Hetzner, `systemctl restart rue-api`, `caddy reload`. Sub-second restart is acceptable (not zero-downtime).

## 13. Observability (v1, minimal)

- `slog` structured logs to stdout, captured by `journalctl`.
- `GET /healthz` endpoint for uptime checks.
- Request ID middleware (`httpx.RequestID`) on every request, included in logs and error responses.
- No metrics/tracing stack — documented as out of scope.

## 14. Security Checklist (v1)

- Paystack webhook HMAC-SHA512 signature verification.
- BetterAuth handles password hashing (argon2).
- All money math server-side; client prices ignored.
- BetterAuth cookie + SameSite=Lax provides CSRF protection.
- CORS origin allowlist via `CORS_ORIGINS` env var.
- Secrets in `.env` (gitignored); `.env.example` ships in the repo.
- Rate limit on `/auth/*` via `httprate`.
- Guest cart tokens are opaque capabilities (no signing); they only authorize cart operations, never identity-bound actions.
